# How to use application mode for bots and CI

**You have:** a CI pipeline, a shared bot, an autonomous agent, or any non-interactive system that needs to talk to Kanboard.

**You want:** that system to act under a service identity, not a human user — so the audit log stays clean, leaving the company doesn't break automation, and per-user quotas don't apply.

**You'll use:** Kanboard's `KANBOARD_AUTH_MODE=app` flag plus an application token.

---

## Why not just use a personal token?

Three problems with using a human's personal token in automation:

1. **Audit confusion** — every comment, task, and attachment created by the bot gets attributed to that human. You can't tell what they did vs what the bot did.
2. **Coupling** — if the human leaves, gets demoted, or rotates their token, automation breaks silently.
3. **Permission drift** — humans have personal permissions that change over time. Service identities have explicit, stable scopes.

Application tokens solve all three. They're issued once, owned by the system not the person, and authored by Kanboard's protocol-level `jsonrpc` user.

## The recipe

### 1. Generate an application token in Kanboard

Log in as a Kanboard admin. Go to **Settings → API → Application token** → **Generate**.

Copy the token. Like personal tokens, it's a 40-character hex string. Store it in your secret manager — never paste it in chat or commit it.

### 2. Configure your client (or CI) for app mode

For an MCP client (Claude Desktop, Cursor, etc.):

```json
{
  "mcpServers": {
    "kanboard": {
      "command": "npx",
      "args": ["-y", "@ernestocorona/kanboard-mcp"],
      "env": {
        "KANBOARD_URL": "https://your-kanboard.example.com",
        "KANBOARD_AUTH_MODE": "app",
        "KANBOARD_API_TOKEN": "the-application-token"
      }
    }
  }
}
```

For CI (GitHub Actions example):

```yaml
- name: Run kanboard automation
  env:
    KANBOARD_URL: ${{ secrets.KANBOARD_URL }}
    KANBOARD_AUTH_MODE: app
    KANBOARD_API_TOKEN: ${{ secrets.KANBOARD_APP_TOKEN }}
  run: npx -y @ernestocorona/kanboard-mcp ./scripts/sync-board.js
```

### 3. Notice what's missing

In app mode, **`KANBOARD_USERNAME` is not required and is ignored**. The server doesn't need it — it uses Kanboard's protocol-level `jsonrpc` user to authenticate every request.

If you accidentally leave `KANBOARD_USERNAME` set, no error — it's just dead config. But if you're switching from personal mode, removing it is a useful signal that you've fully migrated.

## What changes downstream

| Behavior | Personal mode | App mode |
|----------|---------------|----------|
| Author of new tasks/comments | The named user | The `jsonrpc` system user |
| `list_my_tasks` result | Tasks assigned to the named user | Empty (system user owns nothing) |
| `getMe()` at startup | Validates the user exists | Skipped |
| Permission scope | The user's permissions | Admin-equivalent (app token has full API access) |
| Audit log entries | Show the user's name | Show `jsonrpc` |

The big one is the **author**: in app mode, every `create_comment`, `create_task`, `update_subtask`, etc. shows up in Kanboard as authored by `jsonrpc`. That's almost always what you want for automation. If you need it to look like it came from a real user, **don't use app mode** — use a dedicated bot account in personal mode (with its own personal token), and accept that someone has to maintain that account.

## When to use which

| Situation | Recommended mode |
|-----------|------------------|
| Personal use from your own editor | Personal |
| Team-wide CI pipelines | App |
| A shared on-call assistant in Slack | App |
| A long-running bot that posts comments and you want them to look "from the bot" | Personal, with a dedicated bot user account |
| Integration tests in CI | App (keeps the test user's audit log clean) |

## Common issues

| Issue | Likely cause | Fix |
|-------|--------------|-----|
| `401 Unauthorized` | Application token wasn't generated by an admin, or was revoked | Generate a new one in Settings → API |
| Comments show wrong author | You're in app mode and expected your name | Either accept `jsonrpc` as the author (it's by design) or switch to personal mode |
| `list_my_tasks` returns empty | Expected — the system user owns no tasks | Use `list_tasks` with a project filter instead |
| Some action allowed in personal mode is denied in app mode | Application tokens have admin-equivalent scope, but per-project ACLs may still apply | Verify the project's "Public access" / API access settings |

## Security note

Application tokens are powerful. Treat them like SSH deploy keys:

- One token per system. Don't share across CI, Slack bot, and Cron — issue separate tokens so revocation is targeted.
- Rotate on a schedule. Kanboard supports multiple application tokens simultaneously, so you can rotate without downtime.
- Store in a secret manager (GitHub Actions secrets, Vault, AWS Secrets Manager, 1Password Connect). Never in `.env.example`, never in code review.
